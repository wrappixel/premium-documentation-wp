    Authentication in Your Django Project

    Authentication is an essential part of any web application. It ensures only authorized users can access protected areas and perform sensitive operations within the system.

    Authentication System Overview

    MaterialM Django uses Django's built-in django.contrib.auth framework to handle authentication. This powerful module provides:

    • User model for creating and managing users
    • Authentication backend for validating logins
    • Built-in views for login, logout, and password management
    • Forms for handling authentication tasks
    • Permissions & groups support for authorization
    Heads-up: Before you start working with the authentication system, make sure you have created superuser account.

    Authentication Folder Structure

    MaterialM Django comes with a fully organized authentication module containing all the views, templates, and helpers needed for Login, Register, Password Reset, and Email Verification workflows.

    1. Accounts App Structure

    The accounts app manages everything related to authentication. Below is the complete folder structure: 

    
accounts/

  login/
       views.py

  register/
       views.py

  forgot_password/
       views.py

  reset_password/
       views.py

  verify_email/
       views.py

  migrations/

  admin.py
  apps.py
  helpers.py
  models.py
  urls.py
  views.py  (optional for shared logic)

    Each feature (login, register, reset password, etc.) has its own separate module, keeping the authentication logic clean and modular.

    2. Authentication Templates Structure

    All authentication-related HTML templates are stored in: templates/auth/ 

    
templates/

  auth/
        login.html
        register.html
        forgot_password.html
       reset_password.html
       two-step.html

    The templates use MaterialM layout components and extend layouts/blank_page.html for consistent UI styling.

    User Model (Profile)

    MaterialM Django includes a custom Profile model that extends Djangos default User model. It stores additional information such as email verification, tokens, and account metadata.

    1. Profile Model Overview

    Django provides a built-in User model inside django.contrib.auth.models. MaterialM uses a One-to-One linked Profile model to store extra user information. 

    

class Profile(models.Model):
    user = models.OneToOneField(User, on_delete=models.CASCADE, related_name='profile')
    email = models.EmailField(max_length=100, unique=True)
    email_token = models.CharField(max_length=100, blank=True, null=True)
    forget_password_token = models.CharField(max_length=100, blank=True, null=True)
    is_verified = models.BooleanField(default=False)
    created_at = models.DateTimeField(auto_now_add=True)
    otp_expire_at = models.DateTimeField(null=True, blank=True)

    This model allows the system to store email verification tokens, password reset tokens, OTP expiration time, and track whether the user's email has been verified.

    Login

    The Login module provides a secure authentication workflow for users to access the system. Users can log in using their registered email or username along with their password. Once authenticated, the system maintains the user session and redirects them to the Dashboard or the page they were attempting to visit.

    Overview

    The Login screen is built using our custom HTML structure located in templates/auth/login.html. It follows the projects UI standards and includes:

    • Email/Username and Password input fields
    • Error message handling for invalid credentials
    • "Forgot Password" link
    • Support for "next" URL redirection

    View Structure

    The login logic is handled in accounts/login/views.py. The Login View uses LayoutMixin to load the theme layout and mark the active view. The class-based view manages both GET and POST requests, handling user authentication through Djangos built-in authenticate() and login() functions.

    This setup ensures a clean, scalable, and maintainable login system using Djangos built-in authentication framework combined with our UI layout structure.

    Registration

    The Registration page allows new users to create an account in your Django project. This page provides fields for entering basic details such as name, email, username, and password. After filling out the form, users proceed to the next step email verification before the account is fully created.

    Registration Page UI

    The registration page is located at: templates/auth/register.html. It is designed using our clean and modern layout system and includes:

    • Name, Email, Username, and Password fields
    • Real-time validation and error messages
    • Button to submit the registration form
    • Automatic redirection to the Verify Email page

    Handling Registration

    When a user submits the registration form, the information is first saved temporarily in the session. The system does not create the user immediately instead, it sends a verification email to confirm that the email address belongs to the user.

    After this, the user is redirected to the Verify Email page, where they must enter the 6-digit code to complete registration.

    Verify Email

    After the user submits their registration details, a 6-digit verification code is sent to their email address. The user must enter this code on the Verify Email page to complete the registration process. This ensures that only real email owners can activate an account.

    Verify Email Page UI

    The UI for email verification is located at: templates/auth/two-step.html. It includes:

    • Input field for 6-digit verification code
    • Button to submit verification
    • Option to resend OTP
    • Error messages for incorrect or expired OTP

    • Email input field
    • Form validation
    • Success message after sending reset link

    Important Setup Required

    Important INSTALLED_APPS

    These Django apps are required for authentication and messaging. 

    
INSTALLED_APPS = [
    "django.contrib.auth",        # authentication
    "django.contrib.sessions",    # session management
    "django.contrib.messages",    # flash messages
]

    Important MIDDLEWARE

    These are pre-installed in Django projects and required for authentication. 

    
MIDDLEWARE = [

    "django.middleware.csrf.CsrfViewMiddleware",
    "django.contrib.auth.middleware.AuthenticationMiddleware",
    "django.contrib.messages.middleware.MessageMiddleware",

    Important Email Settings

    Use SMTP settings provided by your email service provider. Below is an example for Gmail SMTP. 

    
EMAIL_BACKEND = "django.core.mail.backends.smtp.EmailBackend"

    Make sure to follow the official Gmail or SMTP provider documentation when configuring your settings.

SESSION_COOKIE_SAMESITE = "Lax"

LOGIN_URL = "/login/"               # redirect for login-required pages

    How to Protect Routes Based on Authentication

    In our Django template, route protection ensures that only authenticated users can access specific pages. To achieve this, we use Djangos login_required decorator and the LoginRequiredMixin for class-based views. These tools redirect unauthorized users to the login page before allowing access.

    Note: Our template uses both methods of route protection: function-based decorators and class-based mixins.

    1. Function-Based View Protection

    Apply the login_required decorator on any function-based view to protect it from unauthenticated access. 

    
from django.contrib.auth.decorators import login_required

@login_required(login_url="/login/")
def dashboard(request):
    ...

    2. Class-Based View Protection

    For class-based views, our template uses LoginRequiredMixin along with LayoutMixin to handle layout, context, and authentication checks. 

    
from django.contrib.auth.mixins import LoginRequiredMixin
from django.views.generic import CreateView
from core.mixins import LayoutMixin

class TransactionCreateView(LoginRequiredMixin, LayoutMixin, CreateView):
    template_name = "transactions/create.html"
    login_url = "/login/"
    ...

    In this example, LoginRequiredMixin blocks access for unauthorized users, while LayoutMixin handles UI layout, page titles, and active menu states. This ensures a clean and secure structure throughout the project.
